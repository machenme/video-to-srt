# 视频转日语文本（ASR）并行处理架构设计方案

本项目旨在利用 **RTX 5070 Ti (16GB)** 显卡，通过 Python 构建一套高效、稳定的视频转音频、并行语音转文字（ASR）的自动化流水线。

---

## 1. 技术选型与核心依据

### 核心组件
1. **音频提取**：`ffmpeg` / `moviepy`
   * **理由**：直接提取音频流，不进行重编码，速度极快，CPU 损耗极低。
2. **ASR 引擎**：`faster-whisper`
   * **理由**：基于 CTranslate2，相比 OpenAI 原版加速 4 倍以上。支持 `float16` 精度，对日语同音字、敬语具有极高识别率。

### 硬件性能预算 (RTX 5070 Ti 16GB)
* **系统留空**：Windows 约占用 `1.0 - 1.5 GB` 显存。
* **可用显存**：约 `14.5 GB`。
* **模型选择策略**：
  * **方案 A（高精度）**：`large-v3` (单实例约 3.8GB) -> 推荐并发数：**3**
  * **方案 B（高吞吐）**：`medium` (单实例约 2.0GB) -> 推荐并发数：**6**

## 2. 全链路处理流程

整体架构采用**生产者-消费者（Producer-Consumer）模型**，将“视频转音频”与“音频转文字”解耦，防止 CPU/IO 瓶颈与 GPU 算力瓶颈互相卡死。

```
[ 视频文件输入 ] 
       │
       ▼ (CPU / IO 密集型)
┌──────────────────────────────┐
│  Stage 1: 异步音频提取队列    │ --> 提取出 16kHz Mono WAV
└──────────────────────────────┘
       │
       ▼ (存入临时目录 / 内存队列)
┌──────────────────────────────┐
│  Stage 2: GPU 显存动态调度器  │ --> 控制最大并发实例数 (防止 OOM)
└──────────────────────────────┘
       │
       ├───────────────────────┼───────────────────────┐
       ▼ (GPU 密集型)          ▼ (GPU 密集型)          ▼ (GPU 密集型)
┌──────────────┐        ┌──────────────┐        ┌──────────────┐
│  Worker 1    │        │  Worker 2    │        │  Worker 3    │
│(faster-whisper)       │(faster-whisper)       │(faster-whisper)
└──────────────┘        └──────────────┘        └──────────────┘
       │                       │                       │
       └───────────────────────┼───────────────────────┘
                               ▼
┌──────────────────────────────┐
│  Stage 3: 文本后处理与对齐    │ --> 导出标点规范的 Markdown/SRT
└──────────────────────────────┘
```

## 3. 详细设计步骤

### 步骤一：音频预处理（提取与重采样）
* **操作流**：扫描指定输入文件夹中的视频文件（MP4/MKV 等），将其压入任务队列。
* **核心规范**：
  * 提取时直接降采样至 **`16000Hz`**、**单声道（Mono）**、**16bit PCM (WAV)**。
  * **原因**：Whisper 内部处理的标准格式就是 16kHz 单声道。提前在 CPU 端用 ffmpeg 转换，可以规避 ASR 模型内部转换的额外开销，并大幅减小临时文件体积。

### 步骤二：GPU 并发控制中心（调度层）
* **设计核心**：使用 Python `multiprocessing` (配合 `spawn` 模式) 或进程池。由于 Python GIL（全局解释器锁）的存在，多进程能够更好地榨干多核 CPU 以及管理独立的 GPU 显存上下文。
* **显存保护机制**：
  * 引入信号量（Semaphore）控制最大并发数（如 `large-v3` 设为 3）。
  * 每个进程初始化时，指定 `device="cuda"`，`compute_type="float16"`。

### 步骤三：ASR 推理骨干层
* **日语专属优化配置**：
  * `language="ja"`：必须显式指定语言，若让模型自动侦测，长视频开头的静音或哈气声可能导致模型误判为其他语言，进而导致整段乱码。
  * `beam_size=5`：兼顾速度与幻觉抑制的黄金平衡点。
  * `vad_filter=True`：必须开启。利用 VAD（语音活动检测）提前滤除无声片段、呼吸声、背景噪音，这是防止 Whisper 陷入**无限复读机循环（幻觉）**的关键。

### 步骤四：输出与持久化
* **流式写入**：推理过程中，按时间戳分段（Segments）获取转写结果。
* **格式化**：将带有 `start` 和 `end` 时间戳的数据，格式化为带有时间轴的文本、标准的 `.srt` 字幕文件或清洗后的纯文本。

## 4. 终极性能优化（避坑指南）

1. **避免频繁初始化模型**：
   * 绝对不要在接收到新音频时才去 `WhisperModel()`。模型的加载和显存分配极度耗时。
   * 必须在工作进程启动时**一次性加载模型常驻显存**，后续只消费音频路径。

2. **多进程间的显存隔离**：
   * 在 Windows 下，多进程使用 CUDA 可能会由于上下文切换产生轻微损耗，但在 5070 Ti 级别上完全可以接受。
   * 确保单次转写的音频长度不要极端过长（例如连续 5 小时未分段的单音频），由于上下文窗口增加，显存会有动态波动。

3. **终极吞吐量备选方案 (Batched Inference)**：
   * 如果追求极限速度，可以放弃多进程多开模型。
   * 采用 **单实例 + 批处理（Batching）**：即只常驻一个模型，使用 `batched=True` 选项，将单个长音频切碎成 30 秒的多个 Slice，像矩阵一样打包一次性塞给 GPU。这种做法显存利用率最高，算力利用最纯粹。
